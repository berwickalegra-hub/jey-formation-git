<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → backend | login: untrusted email + password; refresh: refresh-cookie bearer; logout: post-session CSRF-checked; me: cookie-bearer auth |
| backend → DB | login looks up user by email; refresh + me re-query for tokenVersion |
| backend → Redis | lockout-store (login) + refresh-lock (refresh) |

## STRIDE Threat Register

| Threat ID | Severity | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|----------|-----------|-------------|-----------------|
| T-1-01 | HIGH | Spoofing | login (timing-based enumeration) | mitigate | dummyBcryptCompare on no-user branch (D-24); identical INVALID_CREDENTIALS for unknown email + wrong password; bcrypt cost 12 match |
| T-1-01 | HIGH | Spoofing | login (credential stuffing) | mitigate | Per-email rate limit 10/15m + lockout 5 fails / 15 min (D-07, D-08); both keyed on lower-cased email |
| T-1-02 | HIGH | Tampering | refresh (concurrent rotation race) | mitigate | acquireRefreshLock — Redis SETNX + Lua compare-and-delete; 409 CONFLICT on contention with retry hint |
| T-1-02 | HIGH | Elevation of Privilege | refresh (stale tokenVersion bypass) | mitigate | DB-side tokenVersion check before issuing new tokens; mismatch → 401 (post-password-change sessions can't refresh) |
| T-1-02 | HIGH | Elevation of Privilege | me (stale JWT bypass) | mitigate | requireAuth re-queries User.tokenVersion every request (middleware/index.ts:65–75) |
| T-1-03 | MEDIUM | Tampering | logout (CSRF logout) | mitigate | verifyCsrf required (D-02); attacker-forced logout blocked |
| T-1-03 | MEDIUM | Tampering | login + refresh | accept | Pre-session (login) and refresh-cookie-bearer (refresh) — CSRF doesn't apply; documented carve-out |
| T-1-04 | MEDIUM | Information Disclosure | login (which credential failed) | mitigate | Same INVALID_CREDENTIALS for "no user" + "wrong password" + emailVerifiedAt comes only AFTER credential match (D-24) |
| T-1-05 | MEDIUM | Repudiation | All 4 routes | mitigate | withRequestContext wraps body — requestId on every log line + x-request-id header |
| T-1-06 | HIGH | Information Disclosure (timing) | login | mitigate | Constant code path: validation → rate-limit → lockout → user lookup → dummy-or-real-bcrypt → response |
| T-1-07 | MEDIUM | Denial of Service | login (lockout DoS by attacker) | accept | An attacker can lock a victim out by entering 5 wrong passwords for their email; mitigation = forgot-password flow which has its own rate limit. Documented limitation. |
| T-1-08 | LOW | Misconfiguration | env-tunable lockout/rate-limit values | mitigate | Defaults baked via process.env.AUTH_*; ROADMAP Plan 05 documents in .env.example |
</threat_model>
